## Executive Summary

EventSheet AI is built on a dangerous contradiction: it tries to automate enterprise-grade event intelligence within a static Excel file using a zero-budget, one-day development MVP framework ($Base44$, $Gemini$, $Codex$).

While professional event operations live or die by the **Banquet Event Order (BEO)** as the absolute source of truth, treating an Excel workbook generated by an LLM as an active operational operating system introduces extreme risks. This evaluation exposes the deep structural, technical, and market vulnerabilities that could cause EventSheet AI to fail, and outlines what must be stripped away to save it.

---

## Critical Findings: Why EventSheet AI Will Fail

### 1. The Real-Time Paradox (AI vs. Excel Architecture)

The product positioning states: *"The workbook is the product. The website is only an input form."* This architecture breaks down completely under real-world pressure.

* A wedding is an evolving operation with constant changes in guest management, seating arrangements, and budget limits.


* If a professional event manager like Omer Sdot needs to make a critical adjustment on-site—such as an unexpected seating change or a last-minute head count update before a banquet cutoff—they will edit the Excel file directly on their device.


* Because Excel cannot communicate upstream without complex integrations, **the AI layer immediately loses context**. The workbook ceases to be an "intelligent system" and instantly reverts to a standard spreadsheet, breaking the core value loop.

### 2. Overengineered Logic & Feature Bloat

The underlying framework assumes the MVP will handle advanced features such as:

* Complex social engineering maps


* Network theory roles (e.g., *Boundary Spanners*, *Central Connectors*)


* The *Leiden Community Detection Algorithm*

* A *Line-of-Sight (LoS) Suppression Model* to calculate seating angles mathematically



Trying to implement these complex features within a 24-hour MVP budget is unrealistic. High-end operators do not need an algorithm to tell them where to seat the parents. They need a system that enforces strict table capacities, tracks dietary restrictions, and handles sudden additions without breaking the layout.

### 3. Missing Operational Logic

The system focuses heavily on guest psychology while overlooking basic venue constraints. Professional operations require space awareness. If the workbook tracks seating assignments but lacks data fields for venue dimensions, table physical diameters, electrical layouts, or strict vendor delivery timelines, it fails as a true BEO. It operates as a guest tracking sheet, not an event operating system.

### 4. Severe Technical & Security Risks

* **Data Overwrite Disasters:** If the user updates the spreadsheet manually and later uses the web form to regenerate the file via AI, the system risks overwriting custom formulas, manual notes, and on-site changes.


* **The Sensitive Data Trap:** Wedding workbooks naturally collect highly protected personal data, including medical food allergies, child identifiers, VIP security preferences, and direct financial records. Storing and processing this data through a basic Excel sheet without strict access control creates compliance and privacy risks.

```
+------------------------------------+------------------------------------+

+------------------------------------+------------------------------------+
| * Strict Guest & Seating Tracking  | * Network Theory & Louvain/Leiden  |

| * Unified BEO / Operational Truth  | * Line-of-Sight Visual Angle Math  |

+------------------------------------+------------------------------------+
```

* **Remove immediately:** Eliminate all automated social network clustering, visibility angle math, and automated behavioral modeling.


* **Keep and fortify:** Prioritize clean table capacity limits, clear vendor payment milestones, accurate dietary tracking, and a highly visible change log sheet to record manual edits.

## MVP Recommendation & Workbook Architecture

To match the 24-hour development target using Base44 and Gemini, the system must shift from an "automated designer" to an **unbreakable, pre-structured template generator**.

### Workbook Architecture (The 5 Mandatory Sheets)

Every sheet must justify its operational value:

1. **`Event_Master`**: Core details, venue limits, contract deadlines, and contact information.


2. **`Guest_Seating`**: A clean data table tracking `Guest_ID`, `Table_Assignment`, `Dietary_Allergies`, and `VIP_Tier`.


3. **`BEO_Timeline`**: A structured timeline mapping setup windows, service times, and ownership.


4. **`Budget_Control`**: Clean tracking comparing baseline estimates, committed funds, and actual variance.


5. **`Change_Log`**: A simple ledger documenting the *Who*, *When*, and *What* of manual edits to maintain version control.



### User Flow

```
[Input Form] ---> [Gemini Blueprint Generation] ---> [Download Structured Excel Document]
                                                              |
[Post-Event Review / Reconciliation] <--- [On-Site Manual Execution & Edits]

```

## Strategic Risk Analysis

### Excel & AI Limitations

* **Formula Fragility:** A single user accidentally dragging a row incorrectly can break standard `XLOOKUP` functions or `Data Validation` rules.


* **AI Context Drift:** LLMs often struggle with precise mathematical balances across large data arrays. Asking an AI to format a large guest matrix without structural constraints frequently introduces overlapping table numbers or double-assigned seats.



### Demo Strategy

**The Trap:** Presenting a fictional scenario highlighting automated social network mapping risks failure if the AI misinterprets family relationship nuances during a live test.
**The Pivot:** Center the demo on **Last-Minute Response Management**. Show how the generated workbook handles a sudden addition of 10 guests 48 hours before the event, automatically highlights table capacity errors, updates food calculations, and logs changes clearly.

## Recommended Next Steps

1. **Enforce Hard Schema Safeguards:** Use the generation script to lock structural columns and formula matrices, leaving only designated data fields open for manual input.
2. **Define Clear Field Controls:** Add validation rules to key fields (such as `Table_Capacity` and `Vendor_Payment_Milestones`) so the file catches manual input errors immediately.


3. **Validate with Core Operators:** Run a test generation with Omer Sdot using an unstructured guest list to verify if the output format functions reliably under real pressure.



Can an LLM generate a clean, highly functional operational workbook in a single day? Yes. But it will succeed by delivering predictable structure and clear layout validation, not by attempting automated psychological engineering.
